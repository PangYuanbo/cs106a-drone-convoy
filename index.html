02 · Introduction

Why drone convoying and why it matters

Convoying in robotics is defined as having one operator control a lead vehicle while followers follow autonomously. It is a common practice for ground mobile vehicles, but rarely explored in drone applications. In this project, we explored deeply how Drone Convoying could be implemented, and we want to investigate how to create robust drone convoying technology in a resource-constraint setting, in the hopes that it will extend to scenarios like inspection, search-and-rescue, or warehouse logistics.

We built a prototype on the cheapest hardware we could find: a $100 Tello drone (self-funded) with a 720p / 30 fps camera, a laptop running ROS 2 Humble as the base station, and Wi-Fi for everything.

⌬

Hardware

  • DJI Tello · $100 (self-funded)
  • Camera · 720p, 30 fps
  • Compute · base-station laptop, perception off-board
  • Network · Tello UDP RC + H.264 video over Wi-Fi
  • OS · ROS 2 Humble (Ubuntu 22.04 / WSL on Windows)

Our Objectives: Autonomous following with safe failure modes

◎

Goal

Stable moderate-speed following using ArUco markers and face tracking, with safe-landing protocols if the visual lock is lost.

✓

Success bar

30 s of continuous autonomous tracking and maintain a safe social distance the entire time.

↗

Robust Control

We have implemented features that allow the follower to anticipate the leader's acceleration, reacquire after temporary occlusion (when the follower can't find the leader), and extend perception to face/person tracking.

03 · Motivation

How this extends beyond a class project

Real-world extensions

The design of vision-based follower drone that doesn't need GPS or expensive sensors could be able to scale in several directions. For example, In disaster response, a convoy of follower drones could trail a human-piloted lead drone into areas with no GPS or cell coverage and collectively map the scene or relay communications. In agriculture, a lead drone flying a survey pattern could have followers at different altitudes carrying different sensors (multispectral, thermal) without needing each one to be independently piloted. In film and media, multi-angle drone shots currently require one pilot per drone, but a convoy system would let a single cinematographer control the formation. The development of convoying capabilities could unlock new applications and reduce costs in any domain where multiple drones are useful but full autonomy is too risky or expensive.

Where this kind of system is useful in the real world

🔍

Search & rescue

After earthquakes or floods, a human operator can fly a lead drone while follower drones spread out to cover more ground with cameras and thermal sensors — without needing a pilot per drone.

🏗

Infrastructure inspection

In an infrastructure inspection scenario, an inspector could fly a leader drone along the structure and have followers record from different angles to complete the inspection. This would cut crew size and keep people away from dangerous spots effectively.

📦

Warehouse & delivery

As in the large warehouses or campus delivery situations, the convoying system allows a fleet of drones to follow a lead vehicle (ground or air) along a route, each carrying a different package. One operator can handle the whole fleet, reducing the need for multiple pilots and enabling more efficient logistics operations.

How well it actually worked

We tested the system across simulation (Gazebo), real ArUco tracking, and real face tracking. Here's what we measured.

✓

What worked

  • Met the 30-second continuous autonomous tracking bar in both ArUco and face mode
  • Same code path ran in simulation (Gazebo) and on the real Tello without changes
  • Drone auto-takes off after confirming target for ~5 seconds, and auto-lands on prolonged loss
  • Swapping between ArUco and face perception requires no controller changes — just launch a different node
△

Key numbers

  • Target distance: ~1.5 m maintained in both modes
  • EMA smoothing: α = 0.4 on pose estimates
  • RC clamps: 12 / 18 / 12 / 25 (fwd / side / vert / yaw)
  • Staleness tiers: fresh → <150 ms replay → 150–600 ms decay (×0.95) → >600 ms hover → >5 s land

Stage 01 / 05

ArUco perception

Detect the printed ArUco marker and turn it into a 6-DoF pose the controller can servo on.
Reading aruco_perception.py · 1.2 s

Run cv2.aruco.detectMarkers on every camera frame. When ID 0 is visible, run solvePnP with the printed tag size and camera intrinsics — that gives a direct metric translation and rotation.

The yaw is extracted from the rotation matrix and packed into the unused orientation.z field so messaging stays cheap.

# aruco_perception.py · corners + tag size → solvePnP → 6-DoF pose
def on_frame(self, frame):
    corners, ids, _ = cv2.aruco.detectMarkers(frame, self.dict)
    if ids is None:
        return                            # tag not visible this frame

    rvec, tvec = cv2.solvePnP(corners, self.tag_size,
                                  self.K, self.dist)

    pose = Pose()
    pose.position.x, pose.position.y, pose.position.z = tvec
    R, _ = cv2.Rodrigues(rvec)
    pose.orientation.z = math.atan2(
        -R[2,0],
        math.sqrt(R[2,1]**2 + R[2,2]**2),
    )
    self.pose_pub.publish(pose)            # → /aruco/pose_3d
# face_perception.py · YOLOv8 box + ReID → center / area pose
def on_frame(self, frame):
    boxes = self.yolo(frame).boxes
    face  = self.select_target(boxes, self.reid)
    if face is None:
        return self.haar_fallback(frame)   # transient YOLO loss

    H, W = frame.shape[:2]
    pose = Pose()
    pose.position.x = (face.cx - W/2) / (W/2)   # horizontal err
    pose.position.y = (face.cy - H/2) / (H/2)   # vertical err
    pose.position.z = face.area                  # depth proxy

    self.pose_pub.publish(pose)            # → /face/pose
# aruco_controller.py · metric pose servo
# e = [−x, z − d_target, −y, −yaw]
def step(self, pose):
    x, y, z   = pose.position.x, pose.position.y, pose.position.z
    yaw_tag   = pose.orientation.z

    vx  = Kp_fwd  * (z - target_dist) + Kd_fwd  * LPF(self.de_fwd)
    vy  = Kp_side * (-x)              + Kd_side * LPF(self.de_x)
    vz  = Kp_z    * (-y)              + Kd_z    * LPF(self.de_y)
    yaw = Kp_yaw  * (-yaw_tag)        + Kd_yaw  * LPF(self.de_yaw)

    # EMA smoothing α = 0.4 · RC clamps 12 / 18 / 12 / 25
    self.publish_twist(clamp(vx, vy, vz, yaw))
# face_controller.py · image-based servo
# e = [x_center, y_center, A_target − A_face]
def step(self, pose):
    err_x    = pose.position.x                  # normalized [-1, 1]
    err_y    = pose.position.y
    err_area = target_area - pose.position.z    # depth proxy

    yaw = Kp_yaw * err_x    + Kd_yaw * LPF(self.de_x)
    vz  = Kp_z   * (-err_y) + Kd_z   * LPF(self.de_y)
    vx  = Kp_fwd * err_area + Kd_fwd * LPF(self.de_area)

    # Holds head near image center · ~1.5 m social distance
    self.publish_twist(clamp(vx, 0.0, vz, yaw))
# state_machine.py · 5-state safety machine
def step(self, pose, dt):
    if   self.state == IDLE:
        if pose and self.tag_seen_for(2.0):
            self.state = ARM
            self.takeoff()                  # → /tello_action

    elif self.state == FOLLOW:
        if pose is None:
            self.lost_t += dt
            if self.lost_t > 0.5:
                self.state = SEARCH
        else:
            self.lost_t = 0.0
            self.controller.step(pose)        # PD + LPF

    elif self.state == SEARCH:
        self.yaw_scan()
        if   pose:               self.state = FOLLOW
        elif self.lost_t > 5.0: self.state = LAND

    elif self.state == LAND:
        self.soft_land()                    # → /tello_action
07 · Control + Safety

The safety state machine

The leader takes off and is controlled by a human. The follower waits until it sees the tag, takes off, locks on at a safe distance, and tracks the leader. If the tag disappears briefly, it hovers and scans. If it's gone for too long, it lands on its own.

01 · IDLE
Idle
Disarmed. Waiting for tag & takeoff.
02 · ARM
Arm
Stable tag for ~2 s. Auto-takeoff.
03 · FOLLOW
Follow
Track the tag at safe distance.
04 · SEARCH
Search
Tag lost < 2 s. Hover, yaw scan, retry.
05 · LAND
Land
Tag lost > 5 s. Soft-land. Disarm.
08 · Failure Modes + Fixes

Problems we ran into and how we fixed them

▲ 01 Wi-Fi drops & network latency Switched from Docker to WSL on Windows so the Tello UDP socket maps directly through the host. Added a four-tier staleness policy: fresh → <150 ms replay → 150–600 ms decay (×0.95) → >600 ms hover → >5 s land. RC commands rate-limited to 10 Hz, above the Tello's 15-second safety watchdog.
▲ 02 ArUco pose flicker Added EMA smoothing (α = 0.4) on the translation component of the pose. Worked well enough that we didn't need a full Kalman filter.
▲ 03 Sudden tracking errors PD control with a low-pass filter on the derivative term cleans up noisy pose estimates before they reach the actuator. Deadbands and clipped safe speeds prevent jittery commands.
▲ 04 Sub-second occlusions Three-substate search machine: IDLE (staleness tiers handle hover), SEARCH (slow yaw at 0.25 rad/s for up to 8 s, sweeping ~115° per side), COOLDOWN (search timed out, wait for tag return or trigger 5 s land timer).
▲ 05 Multi-node Tello I/O latency Single-owner pattern: only tello_direct_io_node opens Tello SDK/UDP ports. Controller stays "pure ROS" — just publishes /cmd_vel and calls /tello_action. A command_lock serializes sends; non-blocking lock acquisition drops overlapping high-rate commands instead of queueing them.
▲ 06 Inconsistent video stream Collapsed the separate aruco_detector node into a single in-process node that reads the Tello frame and runs detection inline. Eliminates ROS topic transport overhead, serialization, scheduling jitter, and memory copies. Uses DJITelloPy for simpler UDP handling.
09 · Implementation Details

What we needed, what we picked, and the trade-offs

Design criteria

For our system to be useful, it had to meet these requirements:

①

Stable following at safe distance

The follower drone must maintain roughly 1.5 m behind the leader without overshooting or oscillating into it. Collision = broken drone = failed project.

②

Graceful failure

If the drone loses sight of the target, it should not fly off randomly. It needs to hover, scan, and land safely if it can't re-acquire.

③

Swappable perception

We wanted at least two tracking modes (ArUco marker and face) that share the same controller, so we can switch without rewriting control code.

④

Run on cheap hardware

We had a $100 Tello and a laptop. No Jetson, no RTK GPS, no LiDAR. Everything has to work over Wi-Fi with a 720p camera.

Key design choices and trade-offs

DecisionWhat we choseAlternativeWhy
Compute location Off-board (laptop) On-board (Jetson) Tello has no companion-computer port. We send video over Wi-Fi and run perception on the laptop.
Primary tracker ArUco marker Full-body pose / GPS ArUco gives 6-DoF metric pose from a single cheap camera with OpenCV — no training data, no GPS.
Controller PD + low-pass filter PID / MPC PD is simple and good enough at low speed. The low-pass filter handles noisy pose estimates. Adding I-term caused wind-up issues on the Tello.
Communication ROS 2 topics Custom sockets ROS 2 gives us pub/sub, launch files, and easy node swapping. Standard in robotics courses and industry.
Face tracking YOLOv8 + Haar fallback MediaPipe / dlib YOLOv8 is fast and accurate. Haar cascade is a lightweight backup when YOLO drops a frame.

How these choices affect real-world engineering qualities

⛑

Robustness

The five-state safety machine means the drone always has a defined behavior, even when things go wrong. Lost the target? Hover and scan. Still lost after 5 seconds? Land. No undefined states.

⚡

Efficiency

ArUco detection runs at 30 Hz on a laptop CPU — no GPU needed. PD control is a few multiplications per cycle. The whole pipeline fits in the Tello's ~15-minute battery window.

🔧

Maintainability

Because perception and control are separate ROS nodes, we can swap in a new tracker (e.g., LiDAR-based) without touching the controller. Each node is a single Python file under 200 lines.

10b · Conclusion

What we ended up with

Drone convoying on commodity hardware is still mostly an open problem; we wanted to know how far we could push it on a $100 Tello and a laptop. The answer: further than we expected. The follower autonomously tracked a moving leader for 30+ seconds at a time at a ~1.5 m safe distance, in both ArUco-marker and face-tracking modes, running the same control code in Gazebo and on the real drone. Every time the visual lock was lost, the five-state safety machine landed the drone — no crashes across any of our trials.

The most reusable piece of the project isn't any single component — it's the boundary between what we're tracking and how we follow it. Perception (ArUco / YOLOv8 / Haar fallback) and control (PD + low-pass filter on a shared Twist interface) are cleanly separated, so swapping the detector is a launch-file change, not a rewrite. That's the abstraction we'd want in any real deployment, whether the next target is a vehicle, a person, or a thermal blob.

✓

What stuck

  • Stable following at ~1.5 m — PD + EMA smoothing held the distance across both modes, both environments
  • Safe failure, every time — the four-tier staleness policy and search state machine landed the drone in every loss scenario we threw at it
  • Perception was hot-swappable — ArUco and face nodes share the same controller interface; switching is a different launch file, nothing more
  • The whole thing fit in $100 of hardware — no Jetson, no GPU, no RTK GPS. Just a Tello, a laptop, and Wi-Fi
  • Sim parity — Gazebo and the physical drone ran the same nodes and topics; what worked in simulation worked on the real flight
△

Where it still breaks

  • Wi-Fi is the bottleneck — every frame and every command crosses the laptop, capping the effective control rate
  • Outdoor wind exceeds the Tello's correction envelope; the PD loop can't compensate fast enough
  • Face-mode depth is noisy — bounding-box area is a soft proxy compared to ArUco's metric 6-DoF pose
  • ArUco needs a printed tag in the world, which is rarely practical outside a lab
  • No obstacle awareness — the follower sees the target and nothing else. A real deployment would need at least a basic safety bubble

What we'd carry into a follow-up project: the perception/control split is worth keeping, the safety state machine is worth keeping, and the off-board compute pattern is worth replacing. Everything else — the controller gains, the staleness tiers, the search behavior — is tunable and should be re-tuned for whatever drone, environment, and target come next.
